# NUXP Frontend & DevOps Tasks

These tasks are independent of the Code Generator and Core Plugin work. They can be assigned to separate agents immediately.

## Task 6: Frontend Design System

**Objective**: Create a premium, dark-mode first design system for the Vue app.

**Inputs**:
-   `shell/src/styles/`: Where styles should live.
-   `web_application_development` (System Prompt Guidance): "Rich Aesthetics", "Dynamic Design".

**Outputs**:
-   `shell/src/styles/variables.css`: Global CSS variables (colors, spacing, typography).
-   `shell/src/components/ui/Button.vue`, `Input.vue`, `Panel.vue`, `Select.vue`.

**Instructions**:
1.  **Define Themes**: Create a `variables.css` with a curated "Flora" color palette (e.g., deep greens, vibrant accents for selection).
2.  **Components**: Build core UI components using **Vanilla CSS** (scoped). Do not use Tailwind unless necessary.
    -   **Button**: Primary, Secondary, Ghost variants. Hover states.
    -   **Panel**: Glassmorphism effect for background panels.
    -   **Input**: Sleek, minimal inputs with focus rings.
3.  **Typography**: Set up `Inter` or `Outfit` fonts globally.

**Acceptance Criteria**:
-   A demo page (`/design`) showcasing all components.
-   The UI feels "premium" and "native" (matches Illustrator dark mode but better).

## Task 7: Frontend Mock Service

**Objective**: implement a mock API layer so UI development can proceed without the C++ plugin running.

**Inputs**:
-   `shell/src/sdk/generated/`: The interfaces generated by Task 2.

**Outputs**:
-   `shell/src/services/MockBridge.ts`.
-   `shell/src/services/mock/AIArtMock.ts`.

**Instructions**:
1.  **Mock Bridge**: Create a class that intercepts `callCpp` calls when `VITE_USE_MOCK=true`.
2.  **Mock Data**:
    -   Implement `GetArtName`, `GetArtBounds`, `GetArtFirstChild` for `AIArtSuite`.
    -   Return realistic fake data (e.g., "Layer 1", bounds `{ top: 100, left: 50... }`).
    -   Simulate async delays (50ms).
3.  **Toggle**: Integrate this into the main `api.ts` so it can be switched via `.env`.

**Acceptance Criteria**:
-   Running `VITE_USE_MOCK=true npm run dev` allows the app to load and display "fake" document data without errors.

## Task 8: CI/CD Pipeline (GitHub Actions)

**Objective**: Automate the build process for both C++ Plugin and Tauri App.

**Inputs**:
-   `plugin/CMakeLists.txt`.
-   `shell/package.json`.
-   `docs/getting-started.md` (Reference for SDK structure).

**Outputs**:
-   `.github/workflows/build.yml`.
-   `.github/workflows/release.yml`.

**Instructions**:
1.  **Build Workflow**:
    -   Trigger on Push/Pull Request.
    -   **Job 1 (Plugin)**: Runs on `macos-latest` and `windows-latest`.
         -   **SDK Setup**: Download SDK from `ILLUSTRATOR_SDK_URL` (Env Var).
             -   *Note*: User handles hosting the zip (e.g., S3 or GitHub Release) to avoid Adobe login hurdles during CI.
         -   **Build**: Run CMake to build `flora.aip`.
         -   **Artifact**: Upload `flora.aip`.
    -   **Job 2 (Frontend)**: Runs on `ubuntu-latest`. Installs Node. Builds `shell`. Lint & Test.
2.  **Release Workflow**:
    -   Trigger on Tag `v*`.
    -   Builds both, zips them together, and creates a GitHub Release.

**Acceptance Criteria**:
-   A valid `.yml` file that passes `actionlint` or similar static analysis.
-   Correct usage of `actions/checkout`, `actions/setup-node`, `cmake`.
